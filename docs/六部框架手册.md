## 1. 概述

### 1.1 设计理念

**六部框架** 是基于 OpenClaw 平台的多 Agent 协作系统，灵感来自中国古代六部官制：

```
┌─────────────────────────────────────────────┐
│           明代六部官制                        │
├─────────────────────────────────────────────┤
│  吏部 - 人事管理     户部 - 财政税收         │
│  礼部 - 教育外交     兵部 - 军事武官         │
│  刑部 - 司法刑狱     工部 - 工程营造         │
└─────────────────────────────────────────────┘
              ↓ 灵感转化
┌─────────────────────────────────────────────┐
│           OpenClaw 六部框架                   │
├─────────────────────────────────────────────┤
│  吏部 - Agent配置    户部 - 资源管理         │
│  礼部 - 文档沟通     兵部 - 技术执行         │
│  刑部 - 审计合规     工部 - 功能开发         │
│                                              │
│  + 司礼监 - 核心调度（总管）                  │
└─────────────────────────────────────────────┘
```

### 1.2 核心价值

| 价值 | 说明 |
|------|------|
| **分权协作** | 各部门职责清晰，互不干扰 |
| **层级管理** | 司礼监统筹，六部执行 |
| **角色扮演** | 增强趣味性和代入感 |
| **高效通信** | 跨 Agent 消息机制 |
| **容错设计** | 心跳兜底、双重通知 |

### 1.3 适用场景

- ✅ 项目管理（多任务并行）
- ✅ 技术开发（前后端分离）
- ✅ 团队协作（角色分工）
- ✅ 自动化运维（监控告警）
- ✅ 知识管理（文档沉淀）

## 4. 通信原则

### 4.1 心跳兜底机制

每个 Agent 每 30 分钟执行心跳检查：

```
心跳流程
    │
    ▼
┌─────────────────────────┐
│ 1. 读取工部频道最新消息   │
│    message action=read   │
│    target=工部频道ID     │
│    limit=5              │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ 2. 判断是否需要回复       │
│    • @本部？            │
│    • 直接提问？          │
│    • 指令？              │
└───────────┬─────────────┘
            │
       ┌────┴────┐
       │         │
       ▼         ▼
   需要回复    无需处理
       │         │
       ▼         ▼
   及时回复   HEARTBEAT_OK
```

**HEARTBEAT.md 示例**：

```markdown
# 心跳任务

## 1. 先读取工部频道最新消息
message action=read, target=channel:*************, limit=5

## 2. 判断是否需要回复
- 如果有 @本部、直接提问、指令 → 及时回复
- 如果无新消息或无需处理 → 回复 HEARTBEAT_OK

## 3. 注意事项
- 不要在回复中包含 HEARTBEAT_OK
- 跨部门通信使用 @ 提及
- 对话结束时备注「请勿回复」
```

### 4.2 @ 通信规则

| 规则 | 说明 |
|------|------|
| **互通消息用 @** | 六部之间互通消息、汇报司礼监时，使用 @ 提及对方 |
| **回复用 @** | 回复时也使用 @ 提及接收方 |
| **对话要结束** | 对话内容已完成时，最后一次发起方须备注「请勿回复」 |
| **收到即止** | 接收方收到「请勿回复」后，不再回复 |

### ⚠️ 正确的 @ 格式

```
┌────────────────────────────────────────────────────────────┐
│  ✅ 推荐格式（最可靠）                                       │
│  <@*************> 兵部张大人，请复命。                 │
│                                                            │
│  ⚠️ 可选格式（需 Discord 自动补全）                          │
│  @兵部 兵部张大人，请复命。                                  │
│                                                            │
│  ❌ 错误格式（不会触发通知）                                 │
│  @ 兵部 兵部张大人...（有空格）                              │
│  @兵部张大人...（不是 Bot 名）                               │
└────────────────────────────────────────────────────────────┘
```

**为什么推荐 `<@用户ID>` 格式？**

| 格式 | 可靠性 | 说明 |
|------|--------|------|
| `<@*************>` | ✅ 100% | 直接使用用户 ID，永不失效 |
| `@兵部` | ⚠️ 依赖 Discord | 需要 Discord 自动补全，可能失败 |
| `@ 兵部` | ❌ 失效 | 有空格，不是有效提及 |

**示例流程**：

```
司礼监: <@*************> 兵部张大人，请修复登录漏洞。
    ↓
兵部: <@*************> 司礼监大人，臣收到，即刻处理。
    ↓
兵部: <@*************> 漏洞已修复，请查验。请勿回复。
    ↓
司礼监: （不再回复）
```

### 4.3 频道响应规则

| 频道 | 响应方式 |
|------|----------|
| 本部门频道、polytalk、朝会 | 无需 @ 即可响应 |
| 其他频道 | 需要 @ 提及才响应 |

### 4.4 双重通知制（论坛协作）

```
┌─────────────────────────────────────────────┐
│  阶段 1：讨论阶段                             │
│  • 陛下在帖子内 @司礼监 讨论问题              │
│  • 司礼监仅在帖内回复陛下                     │
│  • ❌ 不通知发帖人，避免打扰                  │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  阶段 2：决策阶段                             │
│  • 陛下形成明确结论                           │
│  • 陛下指示：「通知XX部」或「让XX部执行」      │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  阶段 3：通知阶段                             │
│  • 司礼监在论坛帖子内回复决策结果             │
│  • 同时在对应部门频道 @通知                   │
│  • ✅ 确保发帖人收到通知                      │
└─────────────────────────────────────────────┘
```

### 4.3 任务调度原则

| 调度方式 | 适用场景 | 调用方法 |
|----------|----------|----------|
| **即时调用** | 单次查询、原子操作、即时复命 | `openclaw agent --agent [部门] -m "旨意" --deliver` |
| **持久会话** | 多轮协作、异步等待、跨部联动 | `sessions_spawn` + `sessions_send` |

**决策流程**：

```
陛下旨意
    ↓
司礼监研判
    ↓
├─ 单次查询/原子操作/即时复命？→ 即时调用
│
└─ 多轮协作/异步等待/跨部联动？→ 持久会话
        ↓
    奏明方案
        ↓
    得旨后执行
```

```json5
// ~/openclaw.json
{
  // 启用 Agent 间通信
  tools: {
    agentToAgent: {
      enabled: true,
      allow: ["silijian", "gong", "bing", "hu", "hr", "li", "xing"]
    },
```

## 6. 注意事项

### 6.1 配置字段必须精确

| ✅ 正确 | ❌ 错误 | 说明 |
|---------|--------|------|
| `allowAgents` | `allowedAgents` | 子 Agent 允许列表 |
| `allowFrom` | `allowList` | 命令权限配置 |
| - | `allowAny` | 此字段不存在 |

### 6.2 频道 ID 验证

```javascript
// ❌ 错误：发送到错误频道
message({
  target: "channel:*************",  // 工部频道
  message: "⚔️ 兵部张大人..."             // 内容是给兵部的！
})

// ✅ 正确：核对频道 ID
// 兵部频道 ID: 1477874039178530857
// 工部频道 ID: 1477874115112206406
message({
  target: "channel:*************",  // 兵部频道
  message: "⚔️ 兵部张大人..."
})
```

## 7. 管理模式

### 7.1 朝会制度

- **频率**：定期（建议每日或每周）
- **地点**：📢-朝会 频道
- **内容**：
  - 各部门提交工作报告
  - 司礼监汇总汇报
  - 陛下下达新指令

### 7.2 论坛协作

- **地点**：📋-polytalk-项目 论坛频道
- **用途**：
  - 项目讨论
  - 决策留档
  - 问题追踪

### 7.3 Cursor 算力协作

| 部门 | 配额 | 说明 |
|------|------|------|
| 工部 | 2次/5h | 开发任务多 |
| 其他 | 1次/5h | 标准配额 |

**申请格式**：
```
[部门] 申请Cursor - 用途/时长/优先级
示例：[工部] 申请Cursor - 开发用户模块/2h/P1
```

## 8. 快速开始

### 8.1 环境准备

```bash
# 1. 确认 OpenClaw 已安装
openclaw --version

# 2. 确认 Codex CLI 已安装（可选，用于代码生成）
codex --version
```

### 8.2 创建工作空间

```bash
# 创建司礼监工作空间
mkdir -p ~/.openclaw/workspace-silijian

# 创建六部工作空间
mkdir -p ~/.openclaw/workspace-{bing,hu,gong,hr,li,xing}
```

### 8.3 创建配置文件

每个工作空间需要以下文件：

| 文件 | 用途 |
|------|------|
| `SOUL.md` | Agent 身份、职责、行为准则 |
| `AGENTS.md` | 多 Agent 框架说明 |
| `TOOLS.md` | 工具使用指南、频道 ID |
| `HEARTBEAT.md` | 心跳任务说明 |

## 9. 常见问题

### Q1: sessions_send 返回 timeout 怎么办？

**原因**：目标 Agent 当前不活跃。

**解决**：消息已排队，目标 Agent 会在下次心跳时收到。这是正常行为。

### Q2: Discord 频道消息发送后其他 Agent 没收到？

**原因**：OpenClaw 会过滤 Agent 间的频道消息。

**解决**：使用 `sessions_send` 进行跨 Agent 通信。

### Q3: 心跳没有触发？

**检查**：
1. `heartbeat` 配置是否为对象格式（不是字符串）
2. Gateway 是否正在运行
3. `HEARTBEAT.md` 是否存在

### Q4: Slash 命令提示未授权？

**解决**：
```json5
{
  commands: {
    allowFrom: {
      discord: ["user:<YOUR_DISCORD_ID>"]
    }
  }
}
```

### Q5: 如何查看 Session Key？

```bash
# 方法1: 使用 sessions_list
# 在司礼监频道发送：列出所有 sessions

# 方法2: 查看文件
ls ~/.openclaw/agents/*/sessions/*.jsonl
```
